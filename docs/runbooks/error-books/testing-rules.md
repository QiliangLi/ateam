# 错题本 — E2E 测试设计

> 适用场景：编写 E2E 测试、验证功能正确性、确保测试覆盖实际使用场景

### 记录规则

- **条目结构**：❌ 错误做法 + ✅ 正确做法，整条控制在 **5 行以内**
- **❌/✅ 写法**：只写核心规则，禁止多层加粗嵌套（加粗最多一层）
- **`>` 注释**：只放归因路径字母和 postmortem 链接，禁止复述场景
- **复犯**：只追加频率标记（🟡×N / 🔴×N）
- **归属**：条目只写本范围内容，跨范围写到对应文件

### DEV-56 测试只检查关键词，不检查内容质量 `🟡×1`

❌ 测试只检查 `expect(logContainer).toContainText(/codex/)`，codex 回复 "你要我改用普通文本格式" 也算通过
✅ 测试必须验证回复内容与问题的相关性。例如：问编程语言，回复必须包含 `语言|python|typescript` 等关键词；问自我介绍，回复必须包含身份相关内容
> 归因路径 C：新场景（E2E 质量测试之前没有规则）

### DEV-57 测试用例设计与实际使用场景不一致 `🟡×1`

❌ 测试 prompt `请 @codex 告诉我...` 直接触发 codex（前端提取 @codex），但测试期望 opus 先执行
✅ 测试 prompt 必须符合实际使用流程。如果测试 A2A，prompt 应该是 `@opus 请 @codex ...`，让 opus 先执行，然后通过 A2A 触发 codex
> 归因路径 C：新场景（前端 @ 提取逻辑与测试假设不一致）

### DEV-58 测试环境状态不干净导致假绿 `🟡×1`

❌ threadMessages 中有之前的消息，新测试只检查关键词，刚好旧消息包含关键词，测试通过但功能有问题
✅ 每个测试前清空消息区域 `await page.locator('#clearBtn').click()`；测试开始时验证 `await expect(logContainer).toBeEmpty()`
> 归因路径 C：新场景（测试隔离规则之前没有明确）

### DEV-59 反思：为什么测试通过但实际使用有 bug？

**根本原因**：
1. 测试只检查"有没有输出"，不检查"输出对不对"
2. 测试用例设计不合理，与实际使用流程不一致
3. 测试环境没有正确隔离

**改进措施**：
1. 新增质量测试（`a2a-quality.spec.js`）检查回复内容
2. 测试用例必须符合实际使用流程
3. 每个测试前清空环境，确保干净状态

> 归因路径 C：新场景（E2E 测试设计规范之前没有成文规则）
