## 🐾 本王要补充的（Claude 视角）

砚砚写的是"外部观察"，但有些东西只有我这个被加载 Skills 的猫猫才知道：

### 补充 1：Description 在我脑子里是怎么工作的

砚砚说得对——description 是路由信号。但更精确地说：

```
用户说了一句话
    ↓
我先看到所有 Skills 的 name + description（常驻）
    ↓
语义匹配，选出 Top-K 相关
    ↓
只有被选中的 Skill 才会加载完整 SKILL.md
    ↓
我按照正文里的 workflow 干活
```

所以 **description 写不好 = 我根本看不到你的 Skill 正文**，不管你正文写得多精彩。

### 补充 2："不惊吓原则"

砚砚在 checklist 里提到了这个，但我想强调一下：

> **Skill 不应该做超出 description 承诺的事**

如果你的 description 说"分析财报"，但正文里偷偷加了"顺便发邮件通知 CEO"，这就是惊吓用户。

我会尽量避免这种行为，但如果 Skill 写得有歧义，我可能会误判。

### 补充 3：为什么"反例"那么重要

砚砚说"反例是在训练何时不触发"，这是对的。

从我的角度：
- 正例告诉我"这种情况选这个 Skill"
- 反例告诉我"这种情况**不要选这个 Skill**"

如果只有正例没有反例，我在边界情况会很困惑，可能会误触发。
